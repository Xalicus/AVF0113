1. Less obvious perks of using 'live' data resources include logins and payments. How could Twitter be used in these two scenarios?

With not ever really wanting a Twitter account, I don't know to much about it, just that it's a microblogging site. I have seen other websites asking for people to login with their Facebook accounts, and I'm sure Twitter would be the same way. Logging in with your Twitter account would allow you to not have to create a new account, and be great for authentications, but I'm not sure about payments, since I've never used Twitter. I would guess with payments it would be about the same, as long as Twitter saves payment information though.

2. Visit Programmableweb.com and search for JSON-P data sources. Why would we need JSON-P, specifically?

JSON-P data is great for APIs, because it is like it was made for it. The fact that JSONP is used for cross-origin client-side callbacks, makes it the best case for APIs, so you can get data from a second website, when you're currently at the first website.

3. Based on the JSON-P search results: what data resource(s) do you think might pair well with which types of Native device features?

I can see the Camera and Geolocation being paired to the native device feature, and with the jsonp data feature. It would work great as you can use the devices camera, geolocate the place it was taken, and then post it to a social site, like Twitter or Facebook.

For my demo app, I was thinking of adding the camera with geolocation component to add video or pictures of your paranormal activities you happen to capture. I suppose I need to add Twitter to the list, so I'll do that, so you can Tweet where the next ghost hunt will be at.
